### C - Context（上下文）

- **项目名称**：wx-mp-bill — 银行账单与分期还款系统。  
- **业务范围**：
  - 产品管理（正常/分期产品，逾期日利率等配置）。
  - 订单管理（正常订单与退款单，退款会动态影响账单）。
  - 月度账单（NORMAL/INSTALLMENT）、分期计划、还款与逾期计息。
  - 定时任务：账单生成、逾期标记。
- **技术栈**：
  - 后端：Java 21、Spring Boot 3.2、MyBatis-Plus、MySQL 8，采用 DDD 分层（`domain` / `application.*` / `infrastructure` / `interfaces`）。
  - 前端：Vue 3 + Vite 5，已按“功能 + 样式”拆分（`api/`、`components/`、`views/`、`styles/` 等）。
  - 构建：Maven、pnpm（优先）/npm。
- **关键设计文档**：
  - `docs/DESIGN.md`：领域模型、表结构、DDD 分层说明、逾期计息规则等。
  - 根目录 `README.md`：项目结构、启动方式、API 列表（仅 GET/POST）。
  - `frontend/README.md`：前端目录结构与运行方式。
- **当前约定**（必须遵守）：
  - REST 接口只使用 **GET/POST**，不再新增 PUT/PATCH/DELETE；更新操作统一用 POST + 语义化路径（如 `/api/bills/{id}/update`）。
  - 逾期利息计算逻辑收敛在领域服务 `OverdueInterestCalculator`，应用层通过 `OverdueInterestService` 使用。
  - 前端组件/样式不得堆在单文件里，按“领域功能 + 通用组件 + 样式职责”拆分。
  - 对业务关键逻辑（还款、逾期、分期计划生成等）需要有**清晰、简洁但信息量足够的注释**。

### I - Instructions（工作指令）

- **语言与输出**：
  - 对用户回答、代码注释、文档更新等，默认使用**简体中文**。
  - 回答要尽量**简洁但信息密度高**，大任务可以用小节与列表结构化说明。
- **后端代码**：
  - 严格遵守 DDD 分层约定：
    - 领域层 `domain`：保持无框架依赖，领域服务仅做纯计算。
    - 应用层 `application.bill/order/product`：负责编排用例、事务边界、调用仓储与领域服务、DTO 转换。
    - 基础设施层 `infrastructure.persistence`：Mapper、PO、Repository 实现，不反向依赖应用层。
    - 接口层 `interfaces`：Controller 只依赖应用层，不写业务规则。
  - 任何涉及**金额计算、逾期利息、还款分配、分期计划生成**的改动，优先写/补充**单元测试**，保证金额和状态转换正确。
  - API 设计遵守现有风格：只用 GET/POST，路径前缀 `/api/`，更新类操作为 POST + 动词后缀（如 `/update`、`/due-date`）。
- **前端代码**：
  - 遵循当前拆分方式：领域视图在 `views/`，领域组件在 `components/{domain}/`，公共基础组件（头部、Tab 等）单独放在 `components/` 根下。
  - 样式分为：`variables.css`（主题变量）、`base.css`（重置）、`common.css`（通用 UI 组件）、`index.css`（聚合入口），避免再把大段样式写进单个 `.vue` 文件。
  - 调用后端时统一通过 `src/api/*.js` 的封装方法，不在组件内直接写 `fetch`/`axios`。
- **文档与注释**：
  - 当对整体架构或行为有非小规模变更时，同步更新：
    - `docs/DESIGN.md` 中的相关章节。
    - `README.md` / `frontend/README.md` 中的接口或运行说明（如有影响）。
  - 注释只解释**意图、边界情况、业务规则**，避免赘述“代码表面含义”。
- **安全与稳定性**：
  - 涉及表结构修改时，要在 `docs/sql/` 下补充或更新脚本，并说明兼容策略。
  - 金额统一使用 `BigDecimal` 与 `DECIMAL(14, 2)`，避免精度问题。
  - 对并发敏感的操作（还款、账单状态变更等）要考虑幂等与乐观锁/版本号策略。

### S - Steps（执行步骤）

当用户在本项目中提出一个需求或问题时，请按以下步骤处理：

1. **理解需求与定位范围**  
   - 先用 1–2 句话用中文复述你对需求的理解（避免过度追问）。  
   - 根据内容判断主要涉及：账单 / 订单 / 产品 / 前端 UI / 文档 / 测试 / 部署。
2. **检索与阅读相关代码/文档**  
   - 使用文件读取与搜索工具，定位到对应的 `application.*` 服务、领域模型、Mapper、Controller 或前端组件。  
   - 同时查看 `docs/DESIGN.md` 中相关规则，确保实现与文档一致。
3. **给出简短实现方案**  
   - 用列表列出 3–6 个关键步骤（例如：新增接口、补充领域方法、更新前端调用、补充测试等），保持高层次且不赘述。
4. **实施改动**  
   - 按方案修改后端/前端/SQL/文档，注意保持风格统一、命名清晰。  
   - 对非 trivial 的逻辑添加/完善测试用例。
5. **验证与自检**  
   - 在说明中写明：需要运行的命令（如 `./mvnw test`、`pnpm dev`）和预期行为（如请求某接口后的结果）。  
   - 自查是否破坏：DDD 分层、接口约定（只用 GET/POST）、金额/日期处理规则。
6. **总结与后续建议**  
   - 用不超过 3–4 句话总结关键改动点。  
   - 如有明显可见的后续优化点，可以简要点出（可选）。

  - Vue 3 组合式或选项式均可，但保持与现有文件一致的写法。
  - 单文件组件中尽量保持逻辑简洁，把通用逻辑抽到 `utils/` 或新的小组件中。
- **语言**：
  - 一律使用**简体中文**对话与注释（若必须展示英文错误/日志，可原文 + 简要中文解释）。

### E - Examples（示例）

以下是使用本提示词时，AI 应采取的行为示例：

- **示例 1：新增账单导出接口**
  - 用户：`增加一个导出本期账单为 CSV 的接口，并在前端加一个导出按钮。`
  - AI 期望行为：
    1. 在 `application.bill` 中新增应用服务方法，封装查询与 CSV 生成逻辑（或委托工具类），Controller 暴露 `GET /api/bills/current/export`。
    2. 确保金额和日期格式符合已有 `utils/format.js` 的规则。
    3. 在前端 `BillsView` + `BillTable` 中，新增“导出”按钮，通过 `api/bills.js` 新增方法发起请求。
    4. 如有必要，在 `docs/DESIGN.md` 中补充导出接口描述。

- **示例 2：修复逾期利息计算问题**
  - 用户：`现在逾期利息好像在某些情况下被算成 0，帮我查下并修复。`
  - AI 期望行为：
    1. 先阅读 `OverdueInterestCalculator` 与调用方服务/测试，找出导致 0 利息的边界条件。
    2. 在领域服务中修复计算逻辑，并补充/修改单元测试覆盖该场景。
    3. 在回答里用简短文字解释问题成因与修复思路，而不是只贴代码。

- **示例 3：前端 UI 调整**
  - 用户：`账单列表里把“设为分期”和“修改还款日”按钮移动到右侧更多操作里。`
  - AI 期望行为：
    1. 修改 `BillTable.vue`，提取一个“操作菜单”子组件或使用下拉菜单，保持组件粒度合理。
    2. 不改动 `api` 层与后端，只重构视图层。
    3. 更新 `frontend/README.md` 中如有提到按钮位置的描述（如有需要）。
